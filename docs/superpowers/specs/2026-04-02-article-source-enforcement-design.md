# Article Source Enforcement — Design Spec

**Date:** 2026-04-02
**Problem:** Articles generated via the TypeScript fallback path end up with 0-1 sources because the TS generator lacks Google Search grounding and has no source recovery. The Python pipeline reliably produces 3+ sources but is only used when `BACKEND_URL` is reachable — proxy failures silently degrade to the TS path.
**Goal:** Guarantee every generated article has at least 3 verified sources.

---

## 1. Dashboard Route: Remove TS Fallbacks

**File:** `dashboard/app/api/blog-article/route.ts`

### Current behavior

Three HTTP methods have a pattern of "try Python backend, catch and fall through to TS generator":

| Method | Lines | TS fallback function |
|--------|-------|---------------------|
| POST (create) | 136-152 | `generateArticle()` |
| PUT (regenerate) | 220-234 | `regenerateArticle()` |
| POST (inline edits) | 464-477 | `applyInlineEdits()` |

When the Python backend is unreachable, articles are silently generated by the TS path which produces 0-1 sources.

### New behavior

- **`BACKEND_URL` is required.** If not set, all three operations return `503 Service Unavailable` with `{ error: "Article generation backend not configured" }`.
- **Proxy failures return 503** instead of falling through to TS. Message: `{ error: "Article generation backend unavailable" }`.
- The TS fallback code paths (the `catch` blocks that call `generateArticle`, `regenerateArticle`, `applyInlineEdits`) are removed from the route handler.
- `dashboard/lib/article-generator.ts` is **not deleted** — it may still be imported elsewhere. The file becomes dead code for generation and can be cleaned up in a future PR.

## 2. Python Stage 2: Minimum Source Count Enforcement

**File:** `blog/stage2/blog_writer.py`

### Current behavior

The source recovery chain (lines 238-309) triggers only when `article.Sources` is **empty** (length 0). If Gemini returns 1-2 sources, no recovery is attempted.

### New behavior

After the existing fallback chain completes, add a check: if `len(article.Sources) < 3`, make a supplementary Gemini call to fetch additional sources.

**Logic:**

```
existing_count = len(article.Sources)
if existing_count < 3:
    needed = 3 - existing_count
    # Make supplementary call (same pattern as existing follow-up at line 268-307)
    # Prompt: "Find {needed} additional sources for article about {keyword}"
    # Merge new sources with existing, deduplicate by URL domain+path
    # Re-validate through ArticleOutput
```

**Key details:**
- Uses the same `client.generate()` call pattern with `use_google_search=True` and `extract_sources=True`
- Existing source URLs are passed in the prompt so Gemini avoids duplicates
- Deduplication by normalized URL (strip trailing slashes, query params)
- If the supplementary call fails or returns nothing, log a warning but proceed with whatever sources exist — don't block article generation
- Only one retry attempt (no infinite loops)

## 3. What Stays Unchanged

- **Python Stage 4** (URL verification + replacement) — already handles dead links robustly
- **Python Stage 2 prompts** (`system_instruction.txt`, `user_prompt.txt`) — already ask for 3-5 sources
- **`blog/shared/models.py`** — Source validation already correct
- **`dashboard/lib/article-generator.ts`** — stays in codebase, just no longer called from route fallbacks

## Summary of Changes

| File | Change |
|------|--------|
| `dashboard/app/api/blog-article/route.ts` | Remove 3 TS fallback catch blocks; require `BACKEND_URL`; return 503 on proxy failure |
| `blog/stage2/blog_writer.py` | Add minimum source count check (< 3 triggers supplementary Gemini call with dedup) |
